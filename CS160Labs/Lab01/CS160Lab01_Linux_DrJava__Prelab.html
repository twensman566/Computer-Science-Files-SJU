The terminal window

Much of your work will be done using a terminal window. This is a window from which you can issue commands to the Linux operating system. The window may have automatically appeared when you logged in. If not, you can create a new terminal window by selecting Terminal from the System Tools sub-menu of the Applications-menu or by clicking the Terminal icon on the tools bar (the gray screen with a “>_” in it).

The terminal window has a prompt, which generally ends in a percent sign (%) or a dollar sign ($) followed by a cursor, which is a solid black box the size of one character when the window is active or an outline box when the window is not active. When the cursor follows the prompt in a terminal window, you can type commands to the operating system, and they will appear at the cursor position. However, the command you type will not work until you press the Enter key, at which time the computer will execute your command.

Go to the terminal window and type ls (the letters ‘l’ and ‘s’), followed by the Enter key. This command tells Linux to show a list of the files in the current working directory. (Note that Linux is case sensitive; you must use the correct case—usually lower—in all Linux commands.) In this case, you will probably see something like the following:

[jdoe@lin3 1 ~]$ ls
Desktop      Music        Templates
Documents    Pictures     Videos
Downloads    Public
[jdoe@lin3 2 ~]$

Your prompt and file list will probably be different the what is illustrated here. Try to account for why your prompt looks as it does before you go on.

Directories

Like most modern operating systems, Linux systems arrange files hierarchically using directories. A directory is a type of file that conceptually contains files, and since directories are themselves files, directories can contain directory files as well. Directories are also called folders after the common icon used to represent them.

In order to access a file, you need to know what directory it's in, and since a directory is also a file, you need to know which directory contains the directory, and so on. Fortunately, there is one directory not contained in any other directory called the root directory, but named “/”. Files in the root directory can be accessed by a name starting with the slash character followed by file name. For example, there is a directory in the root directory named “usr”; it can be accessed by the name “/usr”. Files in other directories can be named by following the name of the containing directory with another slash and then the file name. Consider the following directory name:

   /usr/people/faculty/cs/foobar/CS160

This long name refers to a file or directory named “CS160” in a directory named “foobar” in a directory named “cs” in a directory named “faculty” in a directory named “people” in a directory named “usr” in the root directory. Such a directory name, starting with the slash character, is refered to as the fully qualified domain name or absolute path of a file; The sequence of directories and sub-directories from the root directory to the desired file is called the path. Any file in the Linux file system can be named in this way. Note that Linux and other flavors of Unix use the slash character (‘/’) to separate directories in a file name; Microsoft Windows uses the backslash character (‘\’) instead. URLs for the Internet also use the slash character. Be careful to distinguish the two characters; you will often hear people use the wrong character name. Note also, that Unix has no concept of separate drive (A:, C:, M:, etc.) as in Windows, although the Unix file system is typically spread over multiple devices; it's just that all that is usually transparent to the user.

Every directory automatically contains two special directories. You generally won’t see these, but they are sometimes useful. You can see them if you type ls -a (list all) at a terminal prompt.

  • A directory named “.” (i.e., a single period character) that refers to the containing directory containing itself. It's a convenient way to name the current directory without specifying the full path name.
  • A directory named “..” (i.e., two periods together) refers to the parent directory of the containing directory.

There may be other files and directories with names beginning with the period character in a directory. Collectively, these files are referred to as hidden files, because they're usually invisible; this is the difference between the plain ls command and the ls -a command; the first doesn't show the hidden files; the latter does.

Another useful directory shortcut to know about is the “~” symbol, which is shorthand for a home directory. (‘~” is the tilde character, which is pronounced till-dee; it is usually found at the upper left corner of the keyboard.) Used alone, the tilde character is shorthand for the user's home directory path; used with another user's ID (e.g. ~jqpublic), it is shorthand for the home directory of that user. For example, if a user named jqpublic has a home directory with the path /usr/people/home/jqpublic, we can refer to the user's www-docs file by the name “~jqpublic/www-docs”.

There's also the concept of the current working directory; when you name a file without an initial slash or tilde character, the computer (usually) computes the path relative to the current working directory. You can see the current working directory with the pwd command; when you first open a terminal, the current working directory is your home directory. You can change the current working directory with the cd command; we'll explore that further below. Suppose the current directory is /usr/people/home/jqpublic and that directory contains a file named foobar; we can refer to the file by its fully qualified domain name /usr/people/home/jqpublic/foobar; we can refer to it directly as foobar since it's in the current working directory; or, assuming jqpublic is a user, we can refer to it as ~jqpublic/foobar. Similarly, we can refer to the parent directory of the current directory /usr/people/home as “..”, and if that directory contains a directory named qjprivate, we can refer to it as ../qjprivate rather than typing /usr/people/home/qjprivate.

Part 2: DrJava

DrJava is an easy to use software development environment for writing Java programs. It was created by members of the Computer Science Department at Rice University, and is currently under active development by the JavaPLT group at Rice. DrJava is designed primarily for student use and provides an interactive facility for developing Java code. It includes an editor and debugger, and it incorporates the JUnit testing framework and the javadoc documentation generating tool.

You can download DrJava for use on your home or dorm room system from http://drjava.sourceforge.net/.

Start DrJava by choosing CSBSJU→Computer Science→DrJava from the Applications menu. DrJava's main window is divided into three panels: The largest area to the right is the definitions pane; you write and edit Java programs in the definitions pane. The narrower panel on the left is the open document list; it displays a list of open documents. You can view and edit a particular document by selecting it from the list; when you click on a document in the list, it will be displayed in the definitions pane. The bottom panel contains several tabbed panes; clicking the tab displays the corresponding pane. For the present, we will be concerned with the interactions pane and the compiler output pane. You can enter and evaluate java statements and expressions in the interactions pane; compilation results are displayed in the compiler output pane.

Configuring DrJava

DrJava should be configured correctly for you automatically, but it's a good idea to check its options before you go on. If the options aren't set correctly, you can have serious problems with compiling your programs. The following steps will insure that DrJava is configured correctly for you.

  • Choose the Preferences option from the Edit pull-down menu; this action will bring up the Preferences window.
  • If it is not already chosen, select the Resource Locations item in the Categories pane on the left. This item allows us to identify resources that DrJava needs to compile and run our programs.
  • Check the text box associated with Tools.jar Location, which is used to let DrJava know where to find the Java program. Verify that it contains a string beginning with /opt/jdk-1.8 and ending with /lib/tools.jar. If not, erase the contents of the text box and enter the string /opt/java-jdk/lib/tools.jar, which will resolve to the correct path.
  • Next, check the text area associated with Extra Classpath. It should contain several directory paths; be sure that it contains the following four, not necessarily in that order:
    • /usr/people/classes/Java
    • /usr/people/classes/Java/acm
    • /usr/people/classes/Java/Javabook/
    • <your home directory path>/JavaPackages
    If any of those paths is missing, use the Add button to add them or ask the instructor or TA for help at the beginning of lab. If there are other paths there, leave them alone.
  • If you made any changes in the previous two steps, click the Apply button to effect the changes.
  • Select the category Miscellaneous. Verify that the item Automatically Close Block Comments is checked. If not, click its check box and then click Apply.
  • Select the category Display Options. Note that there is a check box labeled Show All Line Numbers. If this box is checked, any document displayed in the definitions pane will be shown with numbered lines, which is useful for finding errors; if it is not checked, check it and click Apply.
  • If you wish, click each of the remaining categories to get an idea of the options that can be set.
  • Dismiss the Preferences window by clicking OK.

You don't need to set the options every time you run DrJava. Once you set them, DrJava will remember them the next time you run the application.

The Definitions Pane

Click Open on the DrJava tool bar, and navigate to your Lab01 sub-directory. There should be a file namded TouchyWindow.java in this directory. Select this file and open it. The contents of the file will appear in the definitions pane.

There are several things you should notice about this program. First, the text is displayed in several different colors. Each color denotes a different kind of lexical component. For instance, words like public and class are keywords, which are special words that are predefined in a programming language; DrJava displays all keywords in a unique color. Text in quotes is shown in a different color and numeric values in yet another color. The color coding can help you recognize keying mistakes when you create a program of your own.

Second, the text is indented in a uniform way. This indenting reflects the structure of the program and makes it easier to read. When you key in a program, DrJava automatically indents for you. If the indenting is not what you expect, it is often a sign that you have made a syntactic mistake. Try the following to see how DrJava helps you with indenting:

  • Click at the beginning of the line that begins public void init and enter a few spaces by pressing on the space bar on your keyboard. Do the same with the next couple of lines.
  • Select those lines by dragging the mouse across them so they're highlighted.
  • Press the Tab key. Notice that the original indenting is restored and is once again correct according to Java style standards. (Unlike Python, indenting has no programmatic meaning in Java.)

In DrJava, you can use the Tab key to correct the indenting of the current line or of a group of selected lines. This behavior is unlike most text editors, which insert a Tab character if you press the Tab key. You should also take care not to rely too much on DrJava's indenting prowess; if there are syntax errors in the program, DrJava may not be able to indent it correctly, although that in itself is a good indication that a syntax error is present, usually a mismatched parenthesis or brace.

Third, DrJava will show matching parentheses, braces, and brackets by highlighting the text they enclose.

Compiling a program

You will notice that the document name TouchyWindow.java is followed by an asterisk (*) in the open document list. (You may need to scroll the window or make it wider to see the asterisk.) This symbol means that the document has been modified in the definitions pane but the changes have not been saved to disk. (You modified the document when you added and removed the spaces.)

  • Click the Save button on the tool bar to write the changes to disk. You can also save by choosing Save from the File menu; the Save As choice allows you to save a copy under a different name.

Next, you will compile the class definition; that is, you will invoke a program called a compiler that will translate the Java text program into machine level code that can be interpreted by the Java interpreter. (The interpreter is called the Java Virtual Machine or JVM.) If successful, the compilation process will produce a new file named the same as the Java file but with the extension .class, saved in the current working directory.

  • Click the Compile button on the tool bar.

Something is wrong. Notice that the Compiler Output tabbed pane at the bottom of the window is selected, and it indicates that several errors have been found. The first error is highlighted along with the line in the definitions pane that caused the error.

Let's look carefully at the first error. The error message is at the end of the highlighted part of the Compiler Output pane; it states “class, interface, or enum expected.” That's a pretty cryptic message, and it doesn't really tell you much about what's wrong. (More often, the actual error message states the problem quite clearly.) In this case the problem is pretty obvious; the word “import” is misspelled. Let's correct this problem before going on to the remaining errors.

  • Correct the spelling error in Line 4.
  • Click the Compile button again; (DrJava saves the file automatically when you compile it).

The first error should be gone now, and you'll see two errors in Line 33, followed by two in Line 34; notice also that the indenting appears odd in Lines 34 and 34; (at least they do for the lab writer). It's almost always best to work errors from first to last, because later errors may turn out not to be errors at all, but just fall-out from an earlier error. Let's see if that's the case here. The first error message says “unclosed string literal”, which is a very helpful error message if you know a little Java. Now compare Line 33 that has the error with Line 30 that doesn't have an error. Both lines have very similar syntax, but they're colored differently. How can you account for the difference? In Line 30, the text “Click in this window” is enclosed in straight double-quote characters (not in left and right double quotes as in this write-up); in Line 33, the second quote character is not there. That's what the error message means: text enclosed in double-quote character is called a string literal in Java; the first quote mark opens the literal and the second closes it. We need to add that second quote mark.

  • Add a double-quote character between the ‘!’ and the ‘,’ in Line 33. Notice how the color changes and now matches Line 30.
  • Click the Compile button again.

Now you should still see several errors, but they are different errors that before. This is progress, even if it doesn't seems like it.

One by one we need to correct the errors in the file. The remaining errors in this program should all be fairly easy to figure out, even without much knowledge of Java.

Running a program from DrJava

Java programs are classes that contain a method declared as public static void main, which you can see in Line 54 of the TouchyWindow program. In DrJava, we can run programs with this method by clicking the Run button on the DrJava toolbar. If the program has compiled successfully, the program will now run, although it may or may not run correctly. Programs that don't run correctly is a problem we'll save for another lab!
